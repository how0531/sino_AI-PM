# AIPM 擴充 Agent 建議書

**日期:** 2026-07-14　**依據:** 現有四 Agent 實測經驗(可用連接器、既有連線、踩坑紀錄)

## 現況(已上線)

會議記錄(開單+指派+通知)、時程(查進度/更新/催辦)、KPI(對照目標算缺口)、數據(知識庫查數)。
既有資產:how.kuo 的 Planner/Teams/Outlook/O365使用者 連線都已建好,新 Agent 掛工具**不用再過 OAuth**——這是擴充的最大紅利。

## 建議新增(按 CP 值排序)

### 第五隻:會議排程 Agent (Scheduler)——最推薦

**做什麼**:「幫我約王瑀軒和曹家瑜下週二下午開 30 分鐘會」→ 查與會者行事曆空檔 → 列 2-3 個時段給 PM 選 → 確認後發會議邀請。

| 面向 | 說明 |
|---|---|
| 工具 | Office 365 Outlook 連接器:`尋找會議時間(Find meeting times)`+`建立事件(V4)`,**連線已存在** |
| 名字→email | 沿用「搜尋使用者」模式(已驗證可行) |
| 確認關卡 | 同現有模式:列時段+與會者問一次確認才發邀請 |
| 風險 | 低。發錯會議邀請可取消,可逆 |
| 工時估計 | 半天(照會議記錄 Agent 的既有配方複製) |

**為什麼第一優先**:PM 最花時間的雜事就是喬時間;而且它補完「會議生命週期」閉環——排會(新)→ 開會 → 會議記錄(已有)→ 派單(已有)→ 追蹤(已有)。競賽故事也最完整。

### 第六隻:週報 Agent (Reporter)

**做什麼**:「產本週週報」→ 讀 Planner(本週完成/進行中/逾期)+ KPI 缺口 → 套固定格式產週報草稿 → PM 確認後 Outlook 寄給主管/團隊。

| 面向 | 說明 |
|---|---|
| 工具 | 列出工作(已有配方)+ 傳送電子郵件 V2(已有連線);格式寫死在指示裡 |
| 加值 | 可把「紅黃綠燈」規則寫進指示(逾期>2=紅,1=黃,0=綠) |
| 風險 | 低。寄前必確認 |
| 工時估計 | 半天 |

### 候補(先不做,理由如下)

| Agent | 為什麼先不做 |
|---|---|
| 文件 Agent(SharePoint 找檔/摘要) | 需要先整理 SharePoint 文件庫結構,否則搜出來都是雜訊;等團隊文件歸位再上 |
| 風險 Agent(風險登錄表) | 需要 SharePoint List 當儲存層+PM 維運習慣配合,屬流程變革不是純技術;Phase 2 |
| 請假申請 Agent | 你們已有一隻(簡報截圖那隻),屬 HR 流程,建議獨立不併入 AIPM,避免主 Agent 路由變雜 |

## 現有 Agent 的優化建議(不用新 Agent 就能做)

1. **主 Agent 加「建議的提示」(conversation starters)**:Teams 開啟 AIPM 時顯示 4 顆快捷鈕(貼會議記錄/查逾期/算 KPI/查市佔),新使用者零學習成本。位置:概觀頁「建議的提示」區。
2. **數據 Agent 知識庫月更**:每月電子市佔管線跑完後,把新的 md 檔重新上傳知識庫(取代舊檔)。可寫進電子市佔月更 SOP 最後一步。
3. **會議記錄長逐字稿**:>2000 字建議用測試面板/Teams 的「上傳檔案」傳 txt,避開輸入框上限。
4. **通訊錄查詢擴到時程 Agent**:目前只有會議記錄 Agent 能名字→email;催辦要發同事時還要手給 email。把「搜尋使用者」工具也掛給時程 Agent(10 分鐘,沿用連線)。
5. **路由總量控管**:官方指引主 Agent 可選動作 30-40 個是品質分水嶺;目前 4 子 Agent+工具還很寬裕,加到第 6 隻後建議做一輪路由回歸測試(混淆案例:「約會議討論逾期任務」該去排程還是時程)。
