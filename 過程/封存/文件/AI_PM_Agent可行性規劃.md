# AI PM 專案經理特助｜可行性規劃

> 針對「全方位企業增效 Agent」規劃書的五大 Agent，逐一評估在 Microsoft Copilot Studio 落地的可行性，
> 只做可行的，分兩階段推進。評估日期：2026/07/09。

## 一、可行性總判

判斷標準：**在 Copilot Studio 建得起來、且不踩資安／基礎建設地雷**。
分水嶺是「要不要打公司內部機房的資料庫」——Copilot Studio 在雲端，要連內部
CUBE／神策等系統，必須先架 on-premises data gateway 或把查詢包成內部 API。

| Agent 功能 | 可行性 | 前提 / 卡點 | 階段 |
|---|---|---|---|
| 會議記錄 | ✅ 立即可行 | 純 LLM＋Planner 開單，全在 M365 內 | Phase 1 |
| 提案（PRD 擴充） | ✅ 立即可行 | 純 LLM，一句話→User Story | Phase 1 |
| 時程管理（Planner 版） | ✅ 立即可行 | Planner 原生連接器 | Phase 1 |
| 數據（CUBE／神策） | ⚠️ 有條件可行 | 需 on-prem gateway 或內部 API | Phase 2 |
| KPI 缺口分析 | ⚠️ 有條件可行 | 現況數據吃數據 Agent 的管線 | Phase 2 |
| 提案（競品外網搜尋） | ❌ 先緩 | 幻覺＋對外爬取合規風險 | 剔除 |

**結論：立即做 3 個、有條件 2 個、剔除 1 項。Phase 1 零基建、1–2 週可 demo。**

## 二、逐一評估與風險修正

### 1. 會議記錄 Agent（Phase 1，第一優先）
- **可行**：逐字稿→結論＋待辦→Planner 開單，全程 M365 原生能力。
- **坑**：自動開單 assign 錯人（同名、講錯名字、負責人不明）。
- **修正**：開單前一律 PM 確認（Adaptive Card 關卡）；負責人或期限不明確的待辦標「需 PM 確認」。

### 2. 提案 Agent — PRD 擴充部分（Phase 1）
- **可行**：純 LLM 生成 User Story＋驗收標準，半天可建成。
- **坑**：LLM 憑空假設市場數據。
- **修正**：指令明定「只根據使用者提供的點子推衍」；輸出附「待釐清問題」清單。

### 3. 時程管理 Agent（Phase 1，先做 Planner 版）
- **可行**：Planner 原生連接器撈任務與到期日；Jira 版需另接連接器，列後續。
- **坑**：「智慧催辦」自動私訊是雙面刃——語氣、頻率、誤判（請假中、看板沒更新）都會變成辦公室政治問題。
- **修正**：催辦草稿一律先給 PM 過目、PM 一鍵發送，不自動直發。

### 4. 數據 Agent（Phase 2，價值最高但要基建）
- **有條件可行**：卡點不在串接，在 Text-to-SQL 正確率——LLM 把「次日留存率」翻成錯欄位錯口徑，PM 不會發現，拿錯數字做決策比沒有更糟。
- **修正**：不讓 LLM 自由生 SQL/MDX。預先定義**參數化查詢範本白名單**（留存、漏斗、GMV…約 15 種），LLM 只填參數（日期／平台／功能名）。正確率從「不可控」變「可驗證」，資安面查詢也是白名單。

### 5. KPI 缺口分析 Agent（Phase 2，與數據 Agent 綁定）
- **有條件可行**：目標讀 SharePoint／Excel 容易；現況數據複用數據 Agent 管線。
- **坑**：「策略推薦」LLM 很敢講但沒根據。
- **修正**：Gap 計算（目標 vs 現況 vs 時間進度）用純程式算，數字絕對正確；策略建議明確標「參考方向，需 PM 判斷」。

### 剔除項：競品外網搜尋
內容可能過時或幻覺、對外爬取有合規疑慮。若日後要做，輸出一律標來源與擷取日期、
沒把握的欄位標「未驗證」，定位為「初稿加速器」而非事實來源。

## 三、資料流架構

```
PM 自然語言
  → Copilot Studio（Azure OpenAI，租戶內）
  → Power Automate Flow（帶 PM 身分）
  → [查詢範本白名單]（守門）
  → 內部系統（CUBE／神策／Planner）
  → 回傳 JSON → 繁中摘要＋Markdown 表 → 回覆 PM
  （全程寫稽核 Log）
```

關鍵原則：**LLM 只在入口（理解問題）與出口（轉摘要），中間查詢走白名單，不直接碰資料庫。**

## 四、資安三題（審查前先備好答案）

1. **資料出口**：Copilot Studio（Azure OpenAI）的 prompts 與回應存於 **Microsoft Cloud 信任邊界**內、
   依租戶所選地理區存放，不用於訓練模型、不與 OpenAI 共享；本方案不啟用 Bing 搜尋與第三方外掛。
   （2026/07/10 依官方文件修正措辭，詳見 Copilot_Studio_能力查證報告.md 第 15 項）
2. **權限繼承**：查詢帶 PM 身分做列級權限，不用服務帳號一把撈全部，杜絕越權看數。
3. **稽核軌跡**：誰查了什麼、催了誰、開了什麼單，Power Automate 全程留 Log。

## 五、風險登記（Top 3）

| # | 風險 | 對策 |
|---|---|---|
| 1 | Text-to-SQL 口徑錯，拿錯數字做決策 | 查詢範本白名單，LLM 只填參數 |
| 2 | 自動催辦／自動開單誤傷人 | 一律 PM 確認關卡（人在迴路） |
| 3 | 策略推薦幻覺被當成數據結論 | 數字歸程式、建議標「參考」 |
